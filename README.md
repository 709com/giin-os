# 議員OS ver.1.0 セットアップマニュアル
**文京区議会議員 たかはまなおき（みどりーまん）専用｜2026年7月8日版**

---

## 1. これは何か（全体像）

このキット1つで、議会の仕事が2つの装置に載ります。

```
【装置A：自動巡回・自動配信】GitHub Actions上で無人稼働
  毎朝7時    庁議資料・委員会速報を巡回 → 新着あれば解説記事＋LINE通知
  木曜朝7時  週刊コラム下書き＋バナー画像 → 本人LINEにプレビュー
  （承認ボタン）→ 友だち456人へ一斉配信
  月曜朝7時  選挙情勢レポート（月次。選挙前6ヶ月は自動で週次化）
  第3月曜    財源ハンター（文京区がまだ取っていないお金リスト）

【装置B：質問パイプライン】自分のPCでClaude Codeを開いた時だけ稼働
  資料分析 → 他自治体調査 → 質問文案＋再質問 → 事前公表広報（想定答弁つき）
```

人間（みどりーまん）の仕事は3つだけ：**①資料を入れる ②承認ボタンを押す ③出力を直す**。

## 2. 事前準備（3つの鍵を用意する・計40分）

### 鍵1：Claude APIキー（15分）
1. console.anthropic.com にアクセスしてアカウント作成（claude.aiのサブスクとは別契約）
2. 支払い方法を登録し、まず$10チャージ（月の想定利用は数百〜千円台）
3. API Keys → Create Key → `sk-ant-…` をメモ

### 鍵2：LINEチャネルトークンとユーザーID（20分）
1. manager.line.biz にログイン → たかはまなおきアカウントを選択
2. 設定 → Messaging API → 有効化（プロバイダー名は takahama-office 等で新規作成）
3. developers.line.biz → 該当チャネル → Messaging API設定 → **チャネルアクセストークン（長期）を発行** → メモ
4. 同チャネルの「チャネル基本設定」最下部 → **あなたのユーザーID**（Uから始まる）→ メモ
5. Official Account Managerで料金プランを**ライトプラン（月5,000円税別）**に変更
   ※456人×週1配信＝月約2,000通のため。無料プラン（200通）では足りない
   ※画面名称は変わることがある。迷ったら developers.line.biz/ja/docs/messaging-api/getting-started/

### 鍵3：GitHubアカウント（5分）
1. github.com でアカウント作成（無料でOK）
2. スマホにGitHubアプリを入れる（これが承認ボタンになる）

## 3. 一発セットアップ（30分）

1. **リポジトリ作成**：GitHubで New repository → 名前 `giin-os` → **Public** を選択 → Create
   （Publicにする理由：LINE配信のバナー画像は誰でも見られるURLが必要なため。
   配信物は元々公開情報なので問題なし。非公開メモはこのリポジトリに置かないこと）

2. **キットを投入**：このZIPを展開し、中身を丸ごとリポジトリにアップロード
   （ブラウザなら「uploading an existing file」でフォルダごとドラッグ&ドロップ。
   `.claude` と `.github` は隠しフォルダなので漏れに注意。PCのgitが使えるなら
   `git clone → 中身コピー → git add -A → commit → push` が確実）

3. **鍵を登録**：リポジトリの Settings → Secrets and variables → Actions → New repository secret で3つ登録
   - `ANTHROPIC_API_KEY` ＝ 鍵1
   - `LINE_CHANNEL_TOKEN` ＝ 鍵2のトークン
   - `LINE_USER_ID` ＝ 鍵2のユーザーID

4. **初回起動**：Actions タブ → 「朝の巡回」→ Run workflow
   → 数分後、自分のLINEに「✅巡回を開始しました」が届けば装置Aは稼働開始
   （初回は現状を記録するだけ。翌日以降、新着があった時だけ記事つきで通知が来る）

5. **装置B（質問パイプライン）**：自分のPCで
   ```
   git clone https://github.com/あなたのID/giin-os.git
   cd giin-os
   claude
   ```
   Claude Code内で `/agents` と打ち、4つのエージェントが表示されれば完了。
   使い方：shiryo/ フォルダに委員会資料等を入れて、以下を貼るだけ。
   ```
   テーマ：「（例：フリースクール利用への区独自補助）」
   以下の順で実行。各工程の出力を次工程に引き継ぐこと。
   1. @shiryo-bunseki 2. @tashichi-chosa 3. @shitsumon-sakusei 4. @koho-sakusei
   最後に4ファイルの要点を3行で報告。
   ```

### 週次の承認ルーティン（覚えるのはこれだけ）
- 木曜朝：LINEに下書きが届く → 通勤中に読む
- OKなら：GitHubアプリ → giin-os → Actions → 「承認して配信」→ Run workflow（10秒）
- 直したいなら：GitHubアプリで draft.txt を編集 → その後にボタン

## 6. 改善の重ね方（完成後の育て方）

**原則：プログラムは直さない。プロンプトに1行足す。**

1. **不満メモ方式**：出力に不満を感じたら、その場でLINEのKeepメモ等に1行記録
   （例：「コラムが役所言葉すぎる」「情勢レポートに他会派の動きがない」）
2. **週1・15分のプロンプト手入れ**：金曜にGitHubアプリで該当ファイルを開き、
   「# 厳守事項」に禁止・追加ルールを1行ずつ追記（automation/内の各プロンプト、
   .claude/agents/内の各エージェント）。2〜3週で見違えるほど良くなる
3. **月1のKPIレビュー**（毎月末・15分）：
   - LINE友だち数（目標1,000）とブロック率
   - 巡回の新着検知数と記事の採用率（何本をそのまま使えたか）
   - 答弁トラッカーの「答弁が動いた率」
4. **困ったらこの会話に戻る**：エラー画面のスクショやログを貼れば、私が原因と修正版を出します

## 8. トラブル対応 早見表

| 症状 | 原因の可能性 | 対処 |
|---|---|---|
| 巡回が「失敗しました」通知 | 区サイトのbot対策 | 数日続くようならこの会話に報告。Plan B（既製の更新チェッカー＋手動生成）へ切替設計済み |
| LINEに何も届かない | Secretsの登録ミス | 3つのSecret名のスペルを確認（大文字小文字も一致させる） |
| 配信が失敗する | 通数上限 or トークン期限 | Official Account Managerで残通数確認／トークン再発行 |
| Actionsが動かない | スケジュールの遅延 | GitHubの仕様で数十分遅れることがある。手動Run workflowで代替可 |
| コラムの質が低い | 素材（articles/）が空 | 巡回の新着が無い週は一般テーマで書く仕様。気になるならshiryoに資料を足す |
